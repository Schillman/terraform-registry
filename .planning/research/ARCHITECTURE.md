## Component Boundaries

| Component | Responsibility | Communicates With |
|-----------|---------------|-------------------|
| `modules/{provider}/{resource}/` | Encapsulates one resource abstraction. Public API = variables.tf + outputs.tf | Consumed by example configs and external callers via Git ref |
| `modules/{provider}/{resource}/versions.tf` | Pins provider versions and Terraform minimum version | Terraform CLI reads during `terraform init` |
| `modules/{provider}/{resource}/tests/example/` | Realistic invocation of the module. Doubles as human documentation and CI test fixture | `terraform test`, Terratest |
| `modules/{provider}/{resource}/tests/*.tftest.hcl` | Native Terraform unit/integration assertions | Terraform CLI (`terraform test` command, >=1.6) |
| `modules/{provider}/{resource}/tests/integration_test.go` | Go-based deploy+assert+destroy tests | Terratest library, Go toolchain, real provider API |
| `.github/workflows/test.yaml` | Detects which modules changed, fans out test jobs as matrix | GitHub Actions, Go toolchain, Terraform CLI |
| `.github/workflows/release.yaml` | Calls terraform-module-releaser on main push | terraform-module-releaser GitHub Action |
| `.github/workflows/lint.yaml` | Global quality gate: fmt, tflint, trivy, markdown | super-linter or individual actions |
| `.pre-commit-config.yaml` | Local dev quality gate (mirrors CI) | pre-commit framework |
| `SKILL.md` | Operational guide read by AI agents before any task | Claude, Copilot |

## Canonical Module File List

Every module directory (`modules/{provider}/{resource}/`) MUST contain exactly these files:

| File | Required | Purpose | Notes |
|------|----------|---------|-------|
| `main.tf` | YES | Resource definitions | All `resource` and `data` blocks |
| `variables.tf` | YES | Input variable declarations | All vars typed + described |
| `outputs.tf` | YES | Output value declarations | Required even when empty — terraform-docs and consumers depend on its presence |
| `versions.tf` | YES | `terraform {}` block + `required_providers` | Use `versions.tf` not `terraform.tf` — community standard since TF 0.13 |
| `README.md` | YES | Module documentation | Must contain `<!-- BEGIN_TF_DOCS -->` and `<!-- END_TF_DOCS -->` markers; content auto-generated by terraform-docs |
| `tests/example/main.tf` | YES | Example invocation | Realistic values, not toy examples |
| `tests/example/versions.tf` | YES | Provider config for example | Pins provider version for reproducibility |
| `tests/example/.terraform.lock.hcl` | YES | Provider lock file | Committed for reproducible CI |
| `tests/unit.tftest.hcl` | YES | Native terraform test | At minimum: validates required variables are present and outputs are produced |
| `tests/integration_test.go` | YES | Terratest stub | May be a stub that calls `terraform init/apply/destroy`; expand later |

Files that are OPTIONAL but common:

| File | Purpose |
|------|---------|
| `locals.tf` | Local value computations (keep separate from main.tf when complex) |
| `data.tf` | Data source blocks (separate from resources when numerous) |
| `tests/example/variables.tf` | Expose variables in example so terraform test can pass values |
| `tests/example/outputs.tf` | Expose outputs in example so terraform test can assert values |

## terraform-module-releaser Integration

### How It Detects Modules

**Confidence: MEDIUM** — based on the terraform-module-releaser GitHub Action README (as of mid-2025 training data).

terraform-module-releaser scans the repository for changed `.tf` files and determines which module directory was modified. It derives the release tag from the module path relative to the repo root.

Key constraints:

1. **Path depth is significant.** The tool is designed for `modules/{provider}/{resource}` (3-level paths: `modules/`, provider dir, resource dir). Deeper nesting (e.g., `modules/docker/container/submodule/`) creates tags at unexpected paths and should be avoided.

2. **Tag format is path-derived.** A module at `modules/docker/container/` gets the tag `modules/docker/container/vX.Y.Z`. The consumer `ref` must match this exact tag string.

3. **Conventional Commits drive version bumps.** The commit message prefix determines which semver component increments:
   - `feat:` → minor bump
   - `fix:` → patch bump
   - `BREAKING CHANGE:` in commit footer → major bump
   - `chore:`, `docs:`, `test:` → no release created

4. **Only changed modules are released.** If only `modules/docker/container/` files changed, only that module gets a new tag. Unmodified modules are not tagged.

5. **The `modules/` root directory must not contain `.tf` files directly.** All `.tf` files belong inside a module subdirectory, not at `modules/` level.

### terraform-module-releaser Workflow Placement

```yaml
# .github/workflows/release.yaml
on:
  push:
    branches: [main]
jobs:
  release:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0           # REQUIRED: full history for changelog generation
      - uses: techjavelin/terraform-module-releaser@v1  # verify exact action name
        with:
          github-token: ${{ secrets.GITHUB_TOKEN }}
```

**IMPORTANT:** `fetch-depth: 0` is required. Shallow clones break changelog generation because the releaser needs commit history to determine what changed since the last tag.

## CI/CD Workflow Design: Global vs Per-Module

**Recommendation: Global workflows with matrix fan-out for tests; global for lint and release.**

### Why Not Per-Module Workflows

Storing a workflow file inside each module directory (e.g., `modules/docker/container/.github/workflows/test.yaml`) does not work — GitHub Actions only reads workflows from `.github/workflows/` at the repo root. Per-module workflow files are ignored.

### Three-Workflow Pattern

**Workflow 1: Lint (global, runs on all changed files)**

```yaml
# .github/workflows/lint.yaml
on:
  pull_request:
    branches: [main]
  push:
    branches: [main]
```

Runs: `terraform fmt`, `tflint`, `trivy config`, markdown lint against all changed files. No module awareness needed — linters are recursive.

**Workflow 2: Test (global trigger, matrix fan-out per changed module)**

```yaml
# .github/workflows/test.yaml
on:
  pull_request:
    branches: [main]
  push:
    branches: [main]

jobs:
  detect-modules:
    runs-on: ubuntu-latest
    outputs:
      matrix: ${{ steps.changed.outputs.matrix }}
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0
      - id: changed
        # Script: find changed files, extract unique modules/{provider}/{resource} paths
        # Output: JSON array for matrix
        run: |
          MODULES=$(git diff --name-only HEAD~1 HEAD \
            | grep '^modules/' \
            | awk -F'/' '{print $1"/"$2"/"$3}' \
            | sort -u \
            | jq -R . | jq -sc .)
          echo "matrix={\"module\":$MODULES}" >> $GITHUB_OUTPUT

  test:
    needs: detect-modules
    if: ${{ needs.detect-modules.outputs.matrix != '{"module":[]}' }}
    runs-on: ubuntu-latest
    strategy:
      matrix: ${{ fromJson(needs.detect-modules.outputs.matrix) }}
    steps:
      - uses: actions/checkout@v4
      - uses: hashicorp/setup-terraform@v3
        with:
          terraform_version: "~>1.9"
      - name: terraform test
        run: |
          cd ${{ matrix.module }}/tests/example
          terraform init
          cd ..
          terraform test
        working-directory: ${{ github.workspace }}
      - uses: actions/setup-go@v5
        with:
          go-version: '~1.21'
      - name: Terratest
        run: go test -v -timeout 30m ./...
        working-directory: ${{ github.workspace }}/${{ matrix.module }}/tests
```

**Workflow 3: Release (global, runs on main push after tests pass)**

Calls terraform-module-releaser. See above.

### Path Filter Approach

The detect-modules job uses `git diff` to find changed files and extract the module path prefix (`modules/{provider}/{resource}`). This is more reliable than GitHub Actions `paths:` filters because:
- `paths:` filters at the workflow trigger level — they skip the entire workflow if no matching files changed
- `git diff` at job level lets you run the workflow but skip test jobs when no module files changed
- The matrix correctly fans out one job per changed module

## Test File Placement

### Native Terraform Test (`terraform test`, requires Terraform >= 1.6)

**Location: `modules/{provider}/{resource}/tests/`**

```
modules/docker/container/tests/
├── unit.tftest.hcl          # Tests that do NOT require real infrastructure
└── integration.tftest.hcl   # Tests that DO deploy real infrastructure (optional split)
```

`terraform test` discovers `.tftest.hcl` files in the module directory AND the `tests/` subdirectory automatically. The `tests/` subdirectory is the canonical location when you want to keep test files separate from module files.

Test files reference the example config as their test module root:

```hcl
# tests/unit.tftest.hcl
run "validates_required_variables" {
  command = plan
  module {
    source = "./example"
  }
  variables {
    name              = "test-container"
    docker_image_name = "nginx:latest"
  }
  assert {
    condition     = output.container_name == "test-container"
    error_message = "Container name output did not match input variable"
  }
}
```

**Confidence: HIGH** — Terraform test discovery from `tests/` subdirectory is documented in official HashiCorp docs.

### Terratest (Go)

**Location: `modules/{provider}/{resource}/tests/integration_test.go`**

Place Go test files inside the module's `tests/` directory, NOT at a top-level `testing/` directory. Reasons:

1. Co-location keeps each module self-contained. A developer working on `modules/docker/container/` finds all test assets in one place.
2. Terratest `go test` is run from the `tests/` directory with `go test -v ./...` — it needs to be relative to the module.
3. A top-level `testing/` dir requires import path gymnastics and makes it harder for the matrix workflow to know which test to run for which module.

Go module setup for Terratest:

```
modules/docker/container/tests/
├── go.mod                   # module github.com/Schillman/terraform-registry/modules/docker/container/tests
├── go.sum
├── integration_test.go      # package test
├── unit.tftest.hcl
└── example/
    ├── main.tf
    └── versions.tf
```

**Confidence: MEDIUM** — Co-location is the community convention seen in terraform-aws-modules and similar monorepos. A single top-level `go.mod` is used by some repos but requires path logic in CI.

## Data Flow

### Module Authoring Flow

```
1. Developer creates/modifies files in modules/{provider}/{resource}/
2. Pre-commit hooks fire:
   a. terraform fmt → formats .tf files
   b. terraform-docs → regenerates README.md between markers
   c. tflint → lints Terraform code
   d. trivy config → scans for HIGH/CRITICAL IaC findings
3. Conventional Commit message written (feat:/fix:/chore: prefix)
4. PR opened → CI runs lint.yaml + test.yaml
5. PR merged to main → release.yaml fires terraform-module-releaser
6. Releaser: reads commits since last tag, determines version bump, creates tag
   e.g.: modules/docker/container/v1.2.0
7. Tag pushed to GitHub → consumers can pin ?ref=modules/docker/container/v1.2.0
```

### Module Consumption Flow

```
Consumer Terraform code:
  module "container" {
    source = "git::https://github.com/Schillman/terraform-registry.git//modules/docker/container?depth=1&ref=modules/docker/container/v1.2.0"
  }

  1. Terraform CLI executes git clone with --depth=1 and --branch=modules/docker/container/v1.2.0
  2. Git resolves the tag reference to a commit SHA
  3. Shallow clone fetches only that commit's tree (NOT full history)
  4. Terraform reads modules/docker/container/ directory from the cloned tree
  5. Provider initialization and apply proceed normally
```

## Shallow Clone + Module-Scoped Tags: Gotchas

**Confidence: HIGH** — Based on Git shallow clone behavior and Terraform source protocol documentation.

### How `?depth=1` Works

`depth=1` in a Terraform Git source URL passes `--depth=1` to `git clone`. This fetches only the single commit pointed to by the `ref` tag, without parent history. This is fast and bandwidth-efficient.

### Gotcha 1: Tag Must Be a Lightweight or Annotated Tag Pointing to a Commit

`terraform-module-releaser` creates lightweight tags (standard `git tag`). Lightweight tags work with shallow clones. Annotated tags (created with `git tag -a`) also work. Both resolve to a commit that shallow clone can fetch.

**Risk: None** — terraform-module-releaser creates valid tags that work with `depth=1`.

### Gotcha 2: The Tag Ref Contains Slashes

The ref `modules/docker/container/v1.2.0` contains forward slashes. In Git, tag names may contain slashes and this works correctly in `git clone --branch`. Terraform's Git source handler passes this ref correctly to Git.

**Risk: LOW** — Slash-containing tag names are valid Git refs and Terraform handles them. Validate once during initial migration.

### Gotcha 3: `depth=1` Fetches Only the Tagged Commit's Tree — Not Other Modules

When a consumer clones with `ref=modules/docker/container/v1.2.0`, they get the full tree of the ENTIRE repo at that commit — but only that one commit deep. This means:
- The consumer gets all files in the repo tree at that commit
- Terraform then navigates to `//modules/docker/container` (the double-slash path) within that tree
- Other modules (e.g., `modules/azure/vm/`) are present in the clone but not used

**Risk: None for functionality.** Minor inefficiency: shallow cloning the whole monorepo to use one module means the consumer downloads all module directories, not just the one they need. This is a known tradeoff of monorepo Git sourcing vs. separate repos per module. For small-to-medium monorepos (< 50 modules, < 10MB total), this is acceptable.

### Gotcha 4: `terraform init` Caches the Clone

Terraform caches `.terraform/modules/` locally. The consumer only clones once per `terraform init`. Subsequent `plan`/`apply` use the cached copy. Upgrading requires changing the `ref` in source and re-running `terraform init`.

### Gotcha 5: CI Pipeline Must NOT Use `depth=1` When Running terraform-module-releaser

The releaser workflow requires `fetch-depth: 0` (full history). If you accidentally apply a shallow clone in the release workflow, the releaser cannot read commits since the last tag and will fail or produce incorrect changelogs.

**Prevention:** Keep `fetch-depth: 0` on the release workflow. The lint and test workflows can use `fetch-depth: 1` for speed (they don't need history).

### Gotcha 6: Example Config `source` Must Be Updated After Migration

The existing `tests/example/main.tf` references `github.com/Schillman/terraform-registry//modules/terraform-docker-container?ref=v1.0`. After migration:
- Path changes to `//modules/docker/container`
- Tag format changes from `v1.0` to `modules/docker/container/v1.0.0`
- The example config must be updated BEFORE the first release fires

## Build Order Dependencies

The following order is required when building out the repo structure. Each step has explicit prerequisites.

```
Step 1: Migrate module to namespaced path
  BEFORE: modules/terraform-docker-container/
  AFTER:  modules/docker/container/
  Creates: versions.tf (rename from terraform.tf), outputs.tf (new, empty)
  REQUIRED BY: All subsequent steps — everything depends on correct module path

Step 2: Add README.md markers + terraform-docs config
  Prerequisites: Step 1 (module exists at correct path)
  Creates: terraform-docs markers in README.md
  REQUIRED BY: CI terraform-docs enforcement, terraform-module-releaser wiki generation

Step 3: Add tests/unit.tftest.hcl
  Prerequisites: Step 1 (module with outputs.tf to assert against)
  Creates: tests/unit.tftest.hcl
  REQUIRED BY: Step 6 (test workflow matrix)

Step 4: Add Terratest stub (go.mod + integration_test.go)
  Prerequisites: Step 1 (module path established), Go toolchain available in CI
  Creates: tests/go.mod, tests/go.sum, tests/integration_test.go
  REQUIRED BY: Step 6 (test workflow Terratest job)

Step 5: Update tests/example/main.tf source reference
  Prerequisites: Step 1 (new module path confirmed)
  Modifies: source URL from old flat path to new namespaced path + new tag format
  REQUIRED BY: Step 3 and Step 6 (tests reference the example config)

Step 6: Update/replace CI workflows
  Prerequisites: Steps 1-5 (module structure finalized)
  Creates: release.yaml, test.yaml (replaces/augments lint.yaml)
  REQUIRED BY: Step 7 (releaser cannot run without correct structure)

Step 7: Add .pre-commit-config.yaml
  Prerequisites: Step 2 (terraform-docs markers present), Step 1 (tflint target)
  Creates: .pre-commit-config.yaml
  Can be done: In parallel with Step 6

Step 8: First release
  Prerequisites: Steps 1-7 complete, at least one feat: or fix: commit on main
  Creates: First tag at modules/docker/container/v0.1.0 or v1.0.0
  REQUIRED BY: Consumer source URL migration to new tag format
```

## Architecture Anti-Patterns to Avoid

### Anti-Pattern 1: Flat Module Naming (`terraform-docker-container`)
**What:** Keeping the current `modules/terraform-docker-container/` structure.
**Why bad:** terraform-module-releaser expects `modules/{provider}/{resource}` depth. The flat naming breaks tag format and makes multi-provider repos unnavigable. The tag would be `modules/terraform-docker-container/v1.0.0` — doesn't group by provider.
**Instead:** Migrate to `modules/docker/container/` before wiring up the releaser.

### Anti-Pattern 2: Deeper Than 3-Level Nesting
**What:** `modules/docker/container/v2/` or `modules/docker/container/submodules/network/`
**Why bad:** terraform-module-releaser creates tags based on path detection. Deeply nested paths create unwieldy tag names and break the consumer source URL pattern.
**Instead:** Keep flat: one resource = one directory at `modules/{provider}/{resource}`. If you need submodule variants, use `modules/docker/container-privileged/` (sibling, not nested).

### Anti-Pattern 3: Storing `.tftest.hcl` in Module Root
**What:** `modules/docker/container/test.tftest.hcl`
**Why bad:** Clutters the module's public interface directory. `terraform test` finds these but it mixes test artifacts with the module's consumable files.
**Instead:** All test files belong in `modules/docker/container/tests/`. `terraform test` searches `tests/` subdirectory automatically.

### Anti-Pattern 4: Per-Module Workflow Files
**What:** `modules/docker/container/.github/workflows/test.yaml`
**Why bad:** GitHub Actions only reads workflows from `.github/workflows/` at the repo root. Files in module subdirectories are silently ignored.
**Instead:** Global workflows with matrix fan-out using `git diff` to detect changed modules.

### Anti-Pattern 5: `terraform.tf` Instead of `versions.tf`
**What:** Using `terraform.tf` as the filename for the `terraform {}` block.
**Why bad:** `terraform.tf` is an older convention. The community standard since Terraform 0.13 is `versions.tf`. terraform-docs, tflint, and many tools expect `versions.tf`. It also collides with some tflint rules that expect the `terraform {}` block in a specifically named file.
**Instead:** Rename `terraform.tf` to `versions.tf` during the migration.

### Anti-Pattern 6: Committing `terraform.tfstate` or `.terraform/` in tests/example/
**What:** Accidentally committing Terraform state or provider cache from example runs.
**Why bad:** State files contain sensitive resource IDs and drift quickly. Provider binaries are large and platform-specific.
**Instead:** Ensure `.gitignore` excludes `**/.terraform/`, `*.tfstate`, `*.tfstate.*` (already present in repo).

### Anti-Pattern 7: Using `?depth=1` in CI test workflows
**What:** The test workflow clones with shallow depth when running `terraform test` or Terratest.
**Why bad:** If tests reference the module via Git source URL (as the current example does), a shallow clone in CI cannot resolve tags created in the same run.
**Instead:** Example configs in `tests/example/main.tf` should use a **relative source path** (`source = "../../"`) when running in CI, NOT the published Git URL. The Git URL source is for external consumers, not for the module's own tests.

## Scalability Considerations

| Concern | At 5 modules | At 25 modules | At 100 modules |
|---------|--------------|---------------|----------------|
| CI test time | Single matrix, fast | Matrix parallelism fine | May need test sharding or self-hosted runners |
| `terraform init` shallow clone size | Negligible | Small (all modules cloned) | Consider splitting repo by provider namespace at this scale |
| Release workflow runtime | Fast | Fast (only changed modules tagged) | Fast — terraform-module-releaser is change-driven |
| Directory navigation | Trivial | Clear with provider grouping | Consider a `docs/` index or generated wiki |
| terraform-module-releaser tag count | Low | Manageable | High tag count in GitHub fine; no functional issue |
