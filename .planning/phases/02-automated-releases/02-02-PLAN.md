<interfaces>
<!-- How terraform-module-releaser creates tags -->
- Action triggers: pull_request with types [opened, reopened, synchronize, closed] on main
- Tag format: {relative_module_path}/vX.Y.Z where relative path is from workspace root
- For modules/docker/container/ the tag will be: modules/docker/container/v1.0.0
- Tag is created ONLY when PR is merged (closed event), NOT on push directly to main
- Wiki page is written under the module name (modules/docker/container)

<!-- How PR title validation works -->
- amannn/action-semantic-pull-request validates against conventional commits
- "feat: add initial module" → PASSES
- "did some stuff" → FAILS
- Validation triggers on PR open/edit, not on push to main
</interfaces>

The GitHub Wiki must be initialized manually via the GitHub web UI before the automated release can write to it.

Steps:
1. Open the repository in GitHub: https://github.com/Schillman/terraform-registry (or the actual repo URL from git remote get-url origin)
2. Click the "Wiki" tab
3. Click "Create the first page"
4. Title: "Home"
5. Content:
   ```
   # Terraform Module Registry

   This wiki is auto-generated by terraform-module-releaser. Module documentation is updated automatically on each release.

   ## Modules

   - [Docker Container](modules/docker/container) — managed via automation
   ```
6. Click "Save Page"

After saving, the wiki has at least one page and terraform-module-releaser can write to it on subsequent PR merges.

Why this cannot be automated: GitHub's API does not support wiki initialization on a repo with no existing wiki pages. The first page must be created via the web UI.

<task type="auto">
  <name>Task 2: Create and merge validation PR to trigger first release</name>
  <files>.github/VALIDATION_LOG.md</files>
  <action>
Create a validation PR with a feat: commit to trigger the first automated release.

Step 1 — Create a branch:
```bash
git checkout -b feat/phase-2-validation
```

Step 2 — Create the validation log file (this is the artifact the PR will introduce):
```
.github/VALIDATION_LOG.md
```

Content of .github/VALIDATION_LOG.md:
```markdown
# Phase 2 Validation Log

**Date:** {current date}
**Purpose:** End-to-end validation of automated release pipeline (REL-06)

## Validation Steps

- [ ] feat: commit merged to main via PR
- [ ] Git tag modules/docker/container/v1.0.0 created by automation
- [ ] GitHub Release modules/docker/container/v1.0.0 created with auto-generated notes
- [ ] GitHub Wiki updated with Docker container module page
- [ ] PR title "did some stuff" rejected by CI
- [ ] PR title "feat: add initial module" accepted by CI

## Results

*(populated after merge)*
```

Step 3 — Commit and push:
```bash
git add .github/VALIDATION_LOG.md
git commit -m "feat: add phase 2 validation log"
git push -u origin feat/phase-2-validation
```

Step 4 — Open PR using GitHub CLI:
```bash
gh pr create \
  --title "feat: add phase 2 validation log" \
  --body "Phase 2 automated release validation. Merging this PR should trigger terraform-module-releaser to create tag modules/docker/container/v1.0.0." \
  --base main \
  --head feat/phase-2-validation
```

Step 5 — Wait for CI checks to pass. Check PR status:
```bash
gh pr checks
```

Step 6 — Once all checks pass, merge the PR:
```bash
gh pr merge --merge --delete-branch
```

IMPORTANT: Use `--merge` (not `--squash`) to preserve the conventional commit message. Squash merge works too but ensure the final commit message starts with `feat:`.

Step 7 — Switch back to main and pull:
```bash
git checkout main
git pull
```
